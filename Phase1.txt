# Product Vision

RecallOS is not a recorder.

It is an **AI Memory Operating System**.

Instead of storing raw activities, it stores **understanding**.

Every interaction becomes structured knowledge that can later be reconstructed into human memory.

```
Activity

↓

Context

↓

Meaning

↓

Embeddings

↓

Timeline

↓

Knowledge Graph

↓

Memory
```

# Design Principles

## 1. Local First

Everything works offline.

AI models may run locally.

Cloud AI is optional.

---

## 2. Event Driven

Nothing communicates directly.

Everything emits events.

```
VS Code

↓

Event Bus

↓

Timeline

↓

Search

↓

Analytics

↓

Knowledge Graph

↓

Journal
```

Every new feature simply subscribes to events.

---

## 3. AI Optional

If no AI provider exists

RecallOS still works.

When AI becomes available

It upgrades automatically.

---

## 4. Memory instead of Recording

Never save useless information.

Instead store

```
Problem

↓

Action

↓

Result

↓

Knowledge
```

Example

Instead of

```

Opened VS Code

Opened Terminal
```

Store

```

Read Spring Docs

Fixed Docker Port Conflict

Committed Login Feature
```

# Monorepo Strategy

A monorepo is the right choice because the desktop agent, backend, frontend, shared types, UI library, browser extension, and IDE integrations all need consistent contracts and shared packages. This minimizes duplicated models, keeps APIs synchronized, and simplifies coordinated releases.
