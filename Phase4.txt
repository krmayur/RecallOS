**Status:** ✅ Phase 4 only

No backend implementation yet.

This phase defines the persistent data model that powers RecallOS.

---

# Database Philosophy

RecallOS stores **knowledge**, not raw logs.

Every activity progresses through this lifecycle:

```text
Raw Event
    │
    ▼
Normalized Event
    │
    ▼
AI Summary
    │
    ▼
Embedding
    │
    ▼
Timeline Session
    │
    ▼
Knowledge Graph
```

SQLite stores structured data.

ChromaDB stores semantic vectors.

Large files (screenshots, thumbnails, OCR caches) remain on disk.

# Full-Text Search (FTS5)

SQLite's FTS5 virtual tables will index:

* Event titles
* Event summaries
* OCR text
* Journal content
* AI conversations

This enables instant keyword search before semantic search is used.

---

# Hybrid Search Flow

```text
User Query
    │
    ├──► SQLite FTS5
    │
    └──► ChromaDB
             │
             ▼
      Reciprocal Rank Fusion
             │
             ▼
      Ranked Results
```

Using both approaches provides high precision and good semantic recall.

# Database Migration Strategy

Use a versioned migration system:

```text
001_initial_schema.sql

002_graph_tables.sql

003_ai_tables.sql

004_indexes.sql

005_analytics.sql
```

Each migration is:

* Atomic
* Idempotent
* Rollback-aware where practical

---

# Why this design works for a solo developer

This schema intentionally avoids unnecessary complexity:

* ~20 well-defined tables instead of 60+.
* JSON columns (`metadata_json`) for source-specific data instead of dozens of sparse columns.
* ChromaDB only stores vectors; SQLite remains the authoritative relational store.
* FTS5 provides excellent local keyword search without requiring an external search engine like Elasticsearch.
* The model is extensible but remains understandable by a single engineer.
