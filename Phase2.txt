# RecallOS

# Phase 2 — System Architecture

**Status:** ✅ Phase 2 only

No implementation yet.

This phase defines a production-grade architecture that can scale from a single-user local application to a future multi-device synchronized platform without requiring major rewrites.

---

# Why this architecture?

RecallOS continuously ingests events from many sources (desktop, browser, IDEs, Git, terminal), performs CPU-intensive work (OCR, embeddings, summarization), and serves a responsive UI.

A layered, event-driven architecture isolates responsibilities:

* Capture never waits for AI.
* AI never blocks the UI.
* Storage is replaceable.
* Every module can evolve independently.

# Layered Architecture

```text
Presentation Layer
        │
Application Layer
        │
Domain Layer
        │
Infrastructure Layer
```

---

## 1. Presentation Layer

Contains only UI.

Responsibilities:

* Rendering
* Keyboard shortcuts
* State management
* Search interface
* Timeline visualization
* Graph visualization
* Analytics dashboard

Never contains business logic.

---

## 2. Application Layer

Coordinates workflows.

Example:

```text
User searches

↓

SearchController

↓

SearchService

↓

TimelineService

↓

EmbeddingService

↓

Result
```

Application services orchestrate work.

---

## 3. Domain Layer

Pure business logic.

Contains:

* Event
* Timeline
* Project
* Repository
* Session
* Knowledge Graph
* Search Ranking
* Journal
* Replay

No database code.

No HTTP code.

No Electron code.

---

## 4. Infrastructure Layer

Responsible for:

* SQLite
* ChromaDB
* OCR
* AI providers
* File system
* Electron IPC
* Browser extension communication
* Git commands
* Terminal monitoring

# Electron Architecture

Electron should be thin.

```text
Electron

Main Process

Renderer

Preload

Backend
```

---

## Main Process

Responsibilities

* Window lifecycle
* System tray
* Auto launch
* Native notifications
* Power events
* File dialogs
* IPC bridge

Nothing AI-related.

---

## Renderer

Runs React.

Only renders UI.

---

## Preload

Exposes a secure API.

Example

```typescript
window.recall.search()

window.recall.settings()

window.recall.timeline()
```

No Node.js access from React.

# Security Architecture

Key decisions:

* Context isolation enabled in Electron.
* Node integration disabled in renderer.
* Strict Content Security Policy.
* AES-256 encryption for secrets.
* API keys stored using the operating system's secure credential store where available.
* Parameterized SQL queries only.
* Prompt injection filtering before sending retrieved context to LLMs.
* Per-source permission controls.

# Architectural Decisions Summary

| Decision                   | Reason                                           |
| -------------------------- | ------------------------------------------------ |
| Event-driven design        | Loose coupling and extensibility                 |
| Electron + React           | Mature desktop stack with rich ecosystem         |
| Thin Electron main process | Easier maintenance and security                  |
| Express backend            | Clear separation of UI and services              |
| SQLite + ChromaDB          | Fast structured queries plus semantic search     |
| Worker-based processing    | Keeps UI responsive during AI tasks              |
| Common provider interfaces | Easy switching between AI backends               |
| Hybrid search              | Combines keyword precision with vector relevance |
| Local-first storage        | Privacy, offline support, and user trust         |
| Layered architecture       | Testability and long-term maintainability        |
