# Why a Monorepo?

RecallOS consists of multiple applications that share models, APIs, UI components, and utilities:

* Electron desktop application
* React frontend
* Express backend
* Browser extensions
* IDE integrations
* Shared TypeScript types
* AI provider SDKs
* Database schemas
* Configuration
* Documentation

A monorepo avoids duplicated code and keeps all packages versioned together.

# Dependency Rules

To prevent architectural drift, dependencies are one-way:

```text
Frontend
    │
    ▼
API Client
    │
    ▼
Backend
    │
    ▼
Core Services
    │
    ▼
Repositories
    │
    ▼
Database
```

Rules:

* UI packages must not import database code.
* Backend must not import frontend packages.
* Shared packages must not depend on application packages.
* Domain logic must not depend on Electron.

# Code Standards

* ESLint with zero warnings policy.
* Prettier enforced on commit.
* Conventional Commits for versioning.
* Dependency injection over global state.
* No circular dependencies.
* Comprehensive JSDoc for public APIs.

# CI/CD Pipeline

Every pull request runs:

1. Install dependencies
2. Type checking
3. Linting
4. Unit tests
5. Integration tests
6. Build all packages
7. Electron packaging smoke test
8. Browser extension validation
9. Playwright end-to-end tests
10. Security audit
11. License compliance check

Only fully passing builds can be merged.

# Release Strategy

Independent release channels:

* Desktop Stable
* Desktop Beta
* Backend
* Browser Extension
* VS Code Extension
* Cursor Extension

Semantic versioning is used across the monorepo, with coordinated releases managed by TurboRepo and pnpm workspaces.
